## 1. Описание продукта

Продукт: web/telegram web app/PWA-приложение для учета личных финансов.

Основной сценарий: пользователь быстро добавляет доходы, расходы и переводы, видит состояние счетов, месячный бюджет, аналитику расходов и безопасную сумму, которую можно тратить до конца месяца.

Фокус MVP: приложение должно быть простым для ежедневного учета и при этом помогать пользователю контролировать бюджет.

## 2. Цель проекта

Реализовать рабочий MVP приложения на стеке React + Go.

Результат проекта:

- работающее web-приложение;
- backend API на Go;
- база данных PostgreSQL;
- Docker Compose для локального запуска;
- репозиторий на GitHub;
- GitHub Actions workflow для запуска тестов;
- GitHub Actions workflow для ручного деплоя в Yandex Cloud;
- развернутая версия приложения в Yandex Cloud;
- доска задач в Trello;
- описание API;
- README с инструкцией запуска;
- тесты;
- демонстрационный сценарий.

## 3. Технологический стек

### Frontend

- React;
- TypeScript;

### Backend

- Go;
- PostgreSQL;
- OAuth 2.0 через Yandex ID;
- JWT или server-side session для собственной сессии приложения после входа через Yandex ID;
- Swagger/OpenAPI.

### Инфраструктура

- Docker и Docker Compose;
- GitHub;
- Trello;
- GitHub Actions (CI/CD);
- Yandex Cloud;

### Обязательные инструменты проекта

- Docker используется для локального запуска приложения и базы данных.
- Docker Compose должен запускать frontend, backend и PostgreSQL одной командой.
- GitHub используется для хранения кода, pull requests, code review и истории изменений.
- Trello используется для планирования задач, фиксации статусов и контроля работы команды.
- Задачи из Trello должны соответствовать фактическим pull requests или коммитам в GitHub.
- GitHub Actions должен запускать тесты frontend и backend.
- Деплой в Yandex Cloud должен запускаться вручную по кнопке из GitHub Actions.
- На frontend бизнес-логика не должна размещаться внутри UI-компонентов.
- Backend должен быть построен по clean architecture: бизнес-логика отделена от транспорта, базы данных и внешних сервисов.

### Тестирование

- unit-тесты backend-сервисов;
- API-тесты;
- базовые frontend-тесты;
- ручное end-to-end тестирование ключевых сценариев.

## 7. Критерии приемки MVP

MVP считается готовым, если:

- пользователь может войти через Yandex ID;
- пользователь может создать счета и категории;
- пользователь может добавлять расходы, доходы и переводы;
- балансы пересчитываются корректно;
- дашборд показывает актуальные месячные показатели;
- бюджет показывает план/факт;
- аналитика строится по реальным операциям;
- пользователь может создать регулярный платеж;
- пользователь может создать цель накопления;
- пользователь может экспортировать операции в CSV;
- данные хранятся в PostgreSQL;
- проект запускается через Docker Compose;
- код проекта находится в GitHub;
- командная работа ведется в Trello;
- Trello-задачи связаны с pull requests или коммитами;
- тесты запускаются через GitHub Actions;
- деплой в Yandex Cloud запускается вручную через GitHub Actions;
- backend построен по clean architecture;
- frontend-компоненты не содержат бизнес-логику;
- есть README, миграции и базовые тесты;
- в README описан выбор сборщика и state manager.
