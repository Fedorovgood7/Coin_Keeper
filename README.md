# CoinKeeper — Трекер личных финансов

Веб-приложение для учёта доходов и расходов с бюджетированием, аналитикой и целями накоплений.

## 🚀 Быстрый старт

### 1. Установка Docker

Если Docker не установлен:
- Скачайте Docker Desktop: https://www.docker.com/products/docker-desktop/
- Установите и запустите Docker Desktop
- Дождитесь, пока иконка кита в меню-баре станет стабильной

### 2. Клонирование репозитория

```bash
git clone https://github.com/Fedorovgood7/Coin_Keepeer.git
cd Coin_Keepeer
```

### 3. Настройка переменных окружения

Скопируйте пример файла окружения:

```bash
cp .env.example .env
```

Откройте `.env` и настройте переменные (подробности ниже).

### 4. Запуск приложения

```bash
docker-compose up -d
```

Эта команда запустит:
- **PostgreSQL** на порту 5432
- **Backend API** на порту 8080
- **Frontend** на порту 3000

### 5. Открытие приложения

Откройте браузер и перейдите по адресу: **http://localhost:3000**

Нажмите "Войти через Yandex ID" для входа.

## 🔐 Получение ключей Yandex OAuth

### Шаг 1: Регистрация приложения

1. Перейдите на https://oauth.yandex.ru/client/new
2. Войдите в свой аккаунт Яндекса
3. Нажмите "Зарегистрировать новое приложение"

### Шаг 2: Настройка приложения

1. **Название**: CoinKeeper
2. **Платформы**: выберите "Веб-сервисы"
3. **Redirect URI**: `http://localhost:3000/api/v1/auth/yandex/callback`
4. **Доступы** (в разделе "Яндекс ID"):
   - ✅ `login:email` — получение email
   - ✅ `login:info` — получение имени, аватара
   - ✅ `login:avatar` — получение аватара
5. Нажмите "Зарегистрировать приложение"

### Шаг 3: Получение ключей

После регистрации вы увидите:
- **ID приложения** — это `YANDEX_CLIENT_ID`
- **Пароль приложения** — это `YANDEX_CLIENT_SECRET`

Скопируйте эти значения в файл `.env`.

### Шаг 4: Настройка .env

Откройте `.env` и замените значения:

```env
YANDEX_CLIENT_ID=ваш-id-приложения
YANDEX_CLIENT_SECRET=ваш-пароль-приложения
YANDEX_REDIRECT_URI=http://localhost:3000/api/v1/auth/yandex/callback
JWT_SECRET=случайная-строка-для-jwt
```

Для генерации JWT secret можно использовать:
```bash
openssl rand -base64 32
```

## 🏗 Архитектурные решения

### Выбор сборщика: Vite

**Почему Vite, а не Webpack/CRA:**
- **Скорость**: Vite использует нативные ES-модули браузера, что обеспечивает мгновенный холодный старт dev-сервера (< 300ms vs 10-30s у Webpack)
- **HMR**: Hot Module Replacement работает мгновенно благодаря esbuild
- **Простота**: Минимальная конфигурация, нет сложных конфигов как в Webpack
- **PWA**: Встроенная поддержка через vite-plugin-pwa
- **Современность**: Оптимизирован для современных браузеров с ES modules

### Выбор state manager: Zustand

**Почему Zustand, а не Redux/MobX/Context:**
- **Минимализм**: ~1KB, без бойлерплейта (нет actions, reducers, dispatch)
- **Простота API**: Прямое использование хуков, без провайдеров и обёрток
- **TypeScript**: Отличная поддержка типов из коробки
- **DevTools**: Совместимость с Redux DevTools
- **Persist**: Встроенная middleware для персистентности в localStorage
- **Гибкость**: Можно использовать как глобальный стор или создавать отдельные сторы

### Архитектура Backend: Clean Architecture

```
backend/
├── cmd/app/              # Точка входа
├── internal/
│   ├── application/      # Слой приложения (use cases, DTOs)
│   │   ├── dto/          # Data Transfer Objects
│   │   └── usecase/      # Бизнес-логика по доменам
│   ├── domain/           # Доменный слой (entities, repository interfaces)
│   │   ├── entity/       # Бизнес-модели
│   │   ├── repository/   # Интерфейсы репозиториев
│   │   └── service/      # Доменные сервисы
│   ├── infrastructure/   # Инфраструктурный слой
│   │   ├── config/       # Конфигурация
│   │   ├── postgres/     # Реализации репозиториев
│   │   ├── session/      # JWT-сервис
│   │   └── yandex/       # OAuth-клиент
│   └── presentation/     # Слой представления (HTTP handlers, middleware)
```

**Принципы:**
- Бизнес-логика изолирована от фреймворков и БД
- Зависимости направлены внутрь (domain не зависит от infrastructure)
- Репозитории — интерфейсы в domain, реализации в infrastructure
- Use cases — единственное место для бизнес-логики

### Архитектура Frontend

```
frontend/src/
├── api/          # API-клиент и модули для каждого ресурса
├── services/     # Сервисный слой (facade над API)
├── store/        # Zustand store (состояние + действия)
├── pages/        # Страницы (только UI, без бизнес-логики)
├── types/        # TypeScript типы
└── utils/        # Утилиты
```

**Принципы:**
- UI-компоненты не содержат бизнес-логики
- API-вызовы инкапсулированы в сервисном слое
- Store управляет состоянием и вызывает сервисы
- Типы вынесены в отдельный модуль

## 📊 API Endpoints

### Авторизация
- `POST /api/v1/auth/yandex` — вход через Yandex ID
- `GET /api/v1/auth/me` — получить профиль
- `POST /api/v1/auth/logout` — выход

### Счета
- `GET /api/v1/accounts` — список счетов
- `POST /api/v1/accounts` — создать счёт
- `PATCH /api/v1/accounts/{id}` — обновить счёт
- `PATCH /api/v1/accounts/{id}/archive` — архивировать счёт

### Категории
- `GET /api/v1/categories` — список категорий
- `POST /api/v1/categories` — создать категорию
- `PATCH /api/v1/categories/{id}` — обновить категорию

### Операции
- `GET /api/v1/transactions` — список операций
- `POST /api/v1/transactions` — создать операцию
- `PATCH /api/v1/transactions/{id}` — обновить операцию
- `DELETE /api/v1/transactions/{id}` — удалить операцию

### Бюджет
- `GET /api/v1/budget/monthly` — месячный бюджет
- `POST /api/v1/budget/category-limit` — установить лимит

### Аналитика
- `GET /api/v1/analytics/categories` — расходы по категориям
- `GET /api/v1/analytics/daily` — расходы по дням

### Цели
- `GET /api/v1/goals` — список целей
- `POST /api/v1/goals` — создать цель
- `POST /api/v1/goals/{id}/topup` — пополнить цель

### Экспорт
- `GET /api/v1/export/csv` — экспорт операций в CSV

## 🎬 Демо-сценарий

### Подготовка

1. Запустите проект:
   ```bash
   docker-compose up -d
   ```

2. Откройте http://localhost:3000

3. Войдите через Yandex ID

### Сценарий демонстрации

**Шаг 1: Создание счетов**
1. Перейдите в раздел «Счета»
2. Создайте два счёта:
   - «Наличные» (тип: Наличные, начальный баланс: 5000 ₽)
   - «Основная карта» (тип: Карта, начальный баланс: 50000 ₽)
3. Покажите общий баланс на странице счетов

**Шаг 2: Создание категорий**
1. Перейдите в раздел «Категории»
2. Покажите базовые категории расходов (Еда, Транспорт, Развлечения)
3. Покажите категории доходов (Зарплата, Фриланс)
4. Настройте иконку и цвет для категории «Еда»

**Шаг 3: Добавление дохода**
1. Перейдите на дашборд
2. Нажмите «+ Доход»
3. Выберите:
   - Сумма: 80000
   - Категория: Зарплата
   - Счёт: Основная карта
4. Создайте операцию
5. Покажите обновлённый баланс на дашборде

**Шаг 4: Добавление расходов**
1. Нажмите «+ Расход»
2. Добавьте несколько расходов:
   - 500 ₽, Еда, Наличные
   - 1500 ₽, Транспорт, Основная карта
   - 3000 ₽, Развлечения, Основная карта
3. Перейдите в раздел «Операции» и покажите список
4. Продемонстрируйте фильтрацию по типу (только расходы)

**Шаг 5: Перевод между счетами**
1. Нажмите «Перевод»
2. Переведите 10000 ₽ с Основной карты на Наличные
3. Покажите обновлённые балансы обоих счетов

**Шаг 6: Месячный бюджет**
1. Перейдите в раздел «Бюджет»
2. Установите лимиты по категориям:
   - Еда: 15000 ₽
   - Транспорт: 5000 ₽
   - Развлечения: 10000 ₽
3. Покажите план/факт по каждой категории
4. Покажите безопасную сумму к трате в день

**Шаг 7: Аналитика**
1. Перейдите в раздел «Аналитика»
2. Покажите график расходов по дням
3. Покажите расходы по категориям с процентами
4. Покажите сравнение доходов и расходов

**Шаг 8: Цель накопления**
1. Перейдите в раздел «Бюджет»
2. Создайте цель «Отпуск»:
   - Целевая сумма: 100000 ₽
   - Дедлайн: через 3 месяца
3. Пополните цель на 20000 ₽ с Основной карты
4. Покажите прогресс цели

**Шаг 9: Экспорт**
1. Перейдите в раздел «Операции»
2. Экспортируйте операции в CSV
3. Откройте скачанный файл и покажите данные

**Шаг 10: Дашборд**
1. Вернитесь на дашборд
2. Покажите:
   - Общий баланс
   - Доходы и расходы за месяц
   - Безопасную сумму в день
   - Остаток бюджета
   - Топ категории расходов
   - Регулярные платежи

##  Устранение неполадок

### Порт 3000 уже занят

```bash
lsof -ti:3000 | xargs kill -9
docker-compose up -d
```

### Ошибка подключения к базе данных

```bash
docker-compose ps
docker-compose restart postgres
```

### Ошибка сборки frontend

```bash
cd frontend
rm -rf node_modules
npm install
npm run build
```

### Пересоздание базы данных

```bash
docker-compose down
docker volume rm coinkeeper_postgres_data
docker-compose up -d
```
